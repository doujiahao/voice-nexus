# 智能客服话务平台 — 各团队对接清单

> 本文档列出服务端与各团队的接口对接要点和需要各方适配的变更。
> 接口契约详见：`docs/design-docs/接口契约文档.md`

---

## 一、freeswichService（窦贾浩）

### 他调我们的接口

| 接口 | 方式 | 说明 |
|---|---|---|
| `POST /internal/calls/{fs_call_id}/route` | REST | 呼入路由申请 |
| `POST /internal/calls/{fs_call_id}/events` | REST | 通话结束事件上报 |
| `WS /internal/ws/audio?fs_call_id=xxx` | WebSocket + Protobuf | 音频段上行 |

### 他需要适配的变更

1. **音频上行方式变更**：从 HTTP POST + base64 JSON → WebSocket + Protobuf 二进制
   - 复用现有 `asr.proto` 的 `AudioFrame` 定义
   - 每通电话建立一个 WS 连接，query param 传 `fs_call_id`
   - 握手时 Header 携带 `X-Internal-Key`
   - 对应他的 Task 116（WebSocket 客户端 + 帧封装 + 基础重连）

2. **事件上报路径变更**：从 `POST /v1/call/events` → `POST /internal/calls/{fs_call_id}/events`
   - 请求体格式见接口契约文档 1.2 节

3. **新增路由申请接口调用**：`POST /internal/calls/{fs_call_id}/route`
   - 收到 ESL 呼入事件后，先调此接口获取目标坐席
   - 对应他的 Task 115（FastAPI 4 个核心接口）

### 我们调他的接口（他需要实现）

| 接口 | 方式 | 说明 |
|---|---|---|
| `POST http://freeswich-service/api/v1/calls/action` | REST | 桥接/推流/挂断控制 |

**请求格式**：

```json
// 桥接坐席
{"action": "bridge", "fs_call_id": "xxx", "call_session_id": "xxx", "target_extension": "8001"}

// 开始推流
{"action": "start_streaming", "fs_call_id": "xxx", "call_session_id": "xxx"}

// 挂断
{"action": "hangup", "fs_call_id": "xxx", "call_session_id": "xxx", "hangup_cause": "NORMAL"}
```

**响应格式**：

```json
{"success": true, "fs_call_id": "xxx", "command_result": "OK"}
// 或
{"success": false, "error": "call not found"}
```

> 他目前只实现了 hangup，需要补充 bridge 和 start_streaming。

### 认证方式

- 双向使用 `X-Internal-Key` 静态密钥
- 开发环境 key 值：双方约定一个即可（如 `dev-internal-key-2026`）

## 二、Web 前端（张成）

### 通信方式划分

| 操作类型 | 方式 | 说明 |
|---|---|---|
| 坐席实时操作（接听/挂断/状态切换） | WS 消息 | 带 request_id 应答 |
| 实时事件接收（振铃/转写/分析/结束） | WS 推送 | 服务端主动推送 |
| 查询/管理操作（通话记录/客户/工单） | REST | HTTP 请求 |
| 登录认证 | REST | 使用 JeecgBoot `/sys/login` |

### 他需要适配的变更

1. **WS 消息 type 名变更**：

   | 他现有的 | 改为 |
   |---|---|
   | `incoming_call` | `ringing` |
   | `asr_result` | `transcript` |
   | `call_response` (action: accept/reject) | `call_accept` / `call_reject` / `call_hangup` |
   | `agent_status_update` | `agent_status_change` |

2. **接听/挂断从 WS 简单消息改为带 request_id 应答模式**：
   - 发送时附带 `request_id`（前端生成 UUID）
   - 等待对应的 `_ack` 消息确认成功/失败
   - 超时（如 5s）未收到 ack 视为失败

3. **坐席状态枚举映射**：
   - 前端保持 4 个展示状态：`idle` / `on_call` / `busy` / `offline`
   - 服务端 WS 推送 `agent_status` 时已映射为前端值

4. **REST 接口基础路径**：
   - `VITE_JAVA_API_BASE_URL` 配置为服务端地址
   - `VITE_JAVA_WS_URL` 配置为 `ws://{server}/ws/call-events`

5. **通话记录接口字段对齐**（他已定义的 TypeScript 类型需微调）：
   - `CallListItem.call_session_id` → `id`
   - `CallListItem.duration_ms` → `duration_sec`（单位改为秒）
   - `CallTurnItem` 字段基本一致，微调即可

### WS 连接配置

```
地址: ws://{server_host}:{port}/ws/call-events?token={jwt_token}
心跳: 客户端 20s 发 ping，服务端回 pong
重连: 指数退避，最大 30s，最多 10 次
```

## 三、Call_Handing_Gateway（胡博伟）

### 我们调他的接口

| 接口 | 方式 | 说明 |
|---|---|---|
| `POST /api/v1/asr/transcribe` | REST Multipart | ASR 转写 |
| `POST /api/v1/nlp/intent` | REST JSON | 意图识别 |
| `POST /api/v1/nlp/emotion` | REST JSON | 情绪分析 |
| `POST /api/v1/nlp/entities` | REST JSON | 实体提取 |
| `POST /api/v1/agent-assist/analyze` | REST JSON | 坐席辅助建议 |
| `POST /api/v1/nlp/summarize` | REST JSON | 通话摘要 |

### 他需要适配的变更

1. **改为完全无状态**（他的 Task 101-105 已在做）：
   - 去除 `call_session_id` 等会话字段
   - 上下文由服务端传入，Gateway 不维护

2. **NLP/AgentAssist 接口接收全量 turns**：

   ```json
   {
     "text": "当前句文本",
     "turns": [
       {"role": "customer", "text": "..."},
       {"role": "agent", "text": "..."}
     ],
     "domain": "electric_power"
   }
   ```

   > 注意：他 Task 103 原设计是「当前句 + 最近 N 句」，需要改为接收完整 turns 数组。电网场景通话短，全量传输无性能问题。

3. **ASR 接口接收 WAV 文件**：
   - Multipart 上传
   - 字段：`file`(WAV), `language`("zh"), `speaker_role`, `enable_correction`(bool)
   - 返回：`text`, `corrected_text`, `confidence`, `duration_ms`

4. **摘要接口接收全量 turns**：

   ```json
   {
     "turns": [...],
     "summary_schema": {
       "require_key_points": true,
       "require_pending_actions": true,
       "require_customer_intent": true
     }
   }
   ```

5. **认证方式**：Header `X-Internal-Key`

6. **超时与重试**：
   - 服务端设置 30s 超时
   - 最多重试 3 次（指数退避）
   - 3 次失败后跳过该 turn 的 NLP 分析，不阻塞后续流程

### 接口响应格式要求

所有接口统一返回：

```json
{
  "success": true,
  "data": { ... }
}
// 或
{
  "success": false,
  "error_code": "MODEL_TIMEOUT",
  "message": "模型推理超时"
}
```
